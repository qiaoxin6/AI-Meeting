# AI 运行时故障排障剧本

## 常见现象

- AI 调用超时。
- 同一请求重复执行多次。
- 规则看起来没生效。
- 单次请求拖慢整个链路。

## 先看哪些对象

| 现象 | 先看对象 | 真相源 |
| --- | --- | --- |
| 超时 | `ai-guard.stages.*` | `application.yaml` |
| 重复执行 | `SingleFlight key`、幂等标记 | 代码 + 缓存 |
| 规则没生效 | stage 常量、guard / singleflight 配置 | Java 常量 + YAML |
| 整链路拖慢 | 线程池、限流、恢复范围 | YAML 配置 + 运行时指标 |

## 优先检查

1. 当前 stage 名称是否正确。
2. `ai-guard` 对应 stage 的超时、并发、重试配置。
3. `ai-singleflight` 的 key 是否过细或过粗。
4. `flow-limit` 是否提前把请求挡住。
5. 线程池是否堆积。

## 常见根因

- 新增 stage 没有配置保护策略。
- singleflight key 变更导致去重失效。
- 超时问题其实是文件下载、恢复或下游模型长尾。
- `READ_ONLY` 恢复被误当成运行时异常。

## 处理建议

- 先校准 stage 常量和配置名，再调参数。
- 改 key 前先定义清楚“什么算同一次业务调用”。
- 超时不要只盯模型，要查前置恢复、材料准备和线程池排队。
